▶ MERGE INTO 사용법

MERGE INTO 테이블명  별칭

USING 대상테이블/뷰  별칭

ON 조인조건

WHEN MATCHED THEN
  UPDATE SET
   컬럼1=값1
   컬럼2=값2

WHEN NOT MATCHED THEN
  INSERT (컬럼1,컬럼2,...)
       VALUES(값1,값2,...);

MERGE INTO다음에 나오는 테이블명은 실제로 데이터가 들어가거나 업데이트 되는 테이블이구요.
USING 다음에 나오는 테이블명은 실제 데이터를 가져오거나 할 테이블이구요.
ON은 WHERE과 같은 조건문이죠. 
WHEN MATCHED THEN은 매치되는게 있으면 UPDATE하라는 얘기구요.
WHEN NOT MATCHED THEN은 매치되는게 없으면 INSERT하게 되죠.

[code]
           MERGE INTO INSERTTABLE
           USING DUAL
           ON (ID = 1)
           WHEN MATCHED THEN
           UPDATE SET
           DATA = 'idoori'
           WHEN NOT MATCHED THEN
           INSERT (ID, DATA)
           VALUES (1, 'mudchobo')
[/code]

- 유통으로 발송할 문서들은 규정된 XML로 Packing되어 $HOMS/relay/data/sendtemp 폴더에 저장됩니다.  
- 유통중계 모듈이 정상적으로 동작중이라면 정해진 시간마다 해당 폴더의 XML파일을 읽어서 처리합니다.  
- 만일 처리 중에 오류가 발생하면 유통중계모듈의 로그에 어떤문제로 인해서 처리 오류가 발생했는지가 나타납니다.  

- 그룹웨어 시스템상에서는 상대시스템에서 보내오는 ACK가 있는 경우에만 어떻게 처리되고 있는지를 알수 있습니다.  
- 상대시스템에서 유통 문서를 수신하면 정상처리, 반송, 재전송등의 상태를 ACK로 다시 응답을 보내도록 되어 있습니다.  
- ACK는 receivetemp에 쌓이며 수신중에 오류가 발생한 XML파일은 receiveerr폴더에 저장됩니다.  

- 관리자는 sendtemp나 senderr에 XML파일이 계속 많이 누적되고 있는지를 모니터링 하거나,  
- 유통중계 모듈의 로그를 모니터링 하는 방법으로 정상적으로 유통이 처리되고 있는지를 판단할 수 있습니다.  

- 내용과 같이 sendtemp에 그대로 XML파일이 누적되는 경우는 유통중계모듈에서 정상적으로 처리하고 있지 못한 경우이므로  
- 유통중계 모듈의 로그를 확인해보시고 처리되는 현황을 모니터링해볼 필요가 있습니다.  

=> 수신처 별로 다수의 유통 XML파일이 생성되는 문제는 수정하여 821에 패치하도록 하겠습니다.

-------------------

누락된 부분인 표 아래는 일반 문단이 아니라 표의 하단 캡션으로 작성한 부분입니다.  

현재 아래한글 유통필터가 표의 하단 캡션은 공문서 유통 XML로 변환하지 못하고 SKIP해 버립니다.  

유통 표준에도 표가 table 태그로 변환 시 상단 caption은 가능하지만, 하단 caption은 아예 표준에도 없습니다.  
(실제로 아래한글에서는 상단이던 하단이던 표의 캡션은 유통필터로 변환하지 못합니다. 해당 문서를 HWP로 저장 후,  
  파일-다른 이름으로 저장하기-파일형식-공문서 본문구조로 저장하면 확인 가능)  

따라서 유통 문서 작성 시 표의 캡션 기능을 사용하지 말도록 가이드 해야 합니다.  

공문서 유통 XML은 한글과 같은 워드프로세서의 모든 형식을 지원하지 못하기 때문에, 이와 같이 공문서 유통 XML로  
변환 시 누락되는 부분이 발생할 수 있습니다.  

한컴의 유통필터가 공문서 유통 XML에서 지원하지 못하는 형식이 있다고 오류를 발생시키는 것이 아니라 해당 부분을  
SKIP해 버리는 방식으로 작동하기 때문에, 이와 같이 지원하지 못하는 형식이 문서내에 존재하는 지 검출할 방법이  
현재로써는 존재하지 않습니다.

▶ GW 지원 DBMS

GW 테스트 완료된 환경은 Oracle11g 까지 입니다.

Oracle12c는 아직 테스트 되지 않았습니다.

Oracle 12c 에서는 로컬 사용자 생성이 아닌 공용사용자 생성시에는 사용자 이름을 c## 또는 C## 으로시작해야 합니다.

참고: http://halimdba.blogspot.kr/2013/07/ora-65096-invalid-common-user-or-role.html

ORA-65096: invalid common user or role name or ORA-65049: creation in oracle 12c database
 
 Solution 
 =========================================================================
(you must have to create "common user" under CDB$ROOT with "c##" or "C##" beging of the user name. it is a rule)


아래와 같이 변경해서 계정생성 해보시기 바랍니다.

/* Create New User intraware */
CREATE USER c##intraware IDENTIFIED BY intraware 
	DEFAULT TABLESPACE TSINTRA_1
	TEMPORARY TABLESPACE TSINTRA_T 
	PROFILE DEFAULT;
GRANT "CONNECT" TO c##INTRAWARE;
GRANT "RESOURCE" TO c##INTRAWARE;
GRANT create view TO c##INTRAWARE;

GRANT UNLIMITED TABLESPACE TO c##INTRAWARE;
ALTER USER c##INTRAWARE DEFAULT ROLE ALL;

For MSSql, execute the query from the command prompt as follows:

sqlcmd –S [Server Name] –U [UserID] –P [Password] –i [file path of mssql.sql] -o [output file path of DBVolume.txt]


For example, if Microsoft SQLServer is located on the machine “TKLIS45215”,

and mssql.sql is copied to “c:\temp”, then command line would read as follows.

C:\sqlcmd -S TKLIS45215 -U bizflow -P handy -i C:\temp\mssql.sql -o C:\temp\DBVolume.txt


※ Microsoft SQLServer 가 설치된 서버IP와 Port 로 연결해서 실행하셔도 됩니다.  

사용 예) IP: 127.0.0.1  Port:1433 일 경우.

C:\sqlcmd -S 127.0.0.1,1433 -U bizflow -P handy -i C:\temp\mssql.sql -o C:\temp\DBVolume.txt

▶ 사업기획서

* 외관 보다는 내용에 집중해라
  
 1)왜 하는가? 
기획서를 작성하는 목적에 대한 이유와 필요성이 있어야 함.
  
2)어떻게 할 것인가?
목표를 달성하기 위한 구체적인 방법과 일정, 예산 등을 구체적으로 제시하여야 함.
  
3)실행할 수 있는 가?
현실적으로 가능한 기획인지를 기획서에 분명히 보여줘야 한다.

-----------------------------------------------------------
* 1등 기획서의 특징
 
1)기승전결이 있다.
 핵심과 전략 제시 > 세부적인 실행 계획 > 이를 실천하기 위한 인력, 일정, 자금
  
2)간략하고 명확하다.
 높은 사람에게 제출하는 기획서일수록 1~2쪽이 적당하다. 
 중요한 사항에 대해 필요성과 방법, 그리고 비용 및 기대 효과에 대해서만 간단하고 명확한 메시지만 싣기 때문이다.
  
3)객관적이다.
 기획서는 새로운 수치, 통계, 아이디어를 기반으로 해서 누구나 공감할 수 있는 내용으로 작성해야 한다.
  
4)긍정적이다.
  
5)무슨 내용인지 이해하기 쉽다.
사진, 이미지, 그림, 도표, 그래프 등을 간단하게 이용하여 덧붙여 설명하는 것이 좋다.
  
6)구어체로 작성하되, 가능한 한 명사형으로 종결한다.
 ex) ...을 4단계로 구분합니다. > ...을 4단계로 구분.
  
----------------------------------------------------------
* PDCA 방법으로 기획서를 수정 보완하라
  
1)Plan
 현황 분석 > 목표와의 차이 파악 > 차이 극복 전략 설정 > 세부 실행계획 수립
  
2)Do
 Plan 단계에서 기획한 것을 실행하는 단계이다.
우선순위 결정하고 자원 결정하고 추진 일정 결정, 업무 분장 결정해라. 이때 우선순위를 결정하는 것이 가장 중요하다.
  
3)Check
 앞에 두 단계를 검토하는 단계이다. 체크리스트를 Plan 단계에서 미리 만들어라.
  
4)Action

▶ POS (point of sale) ; 판매시점관리
POS[포스]란 판매와 관련된 데이터를 물품이 판매되는 그 시간과 장소에서 즉시 취득하는 것이다. 

POS 시스템은 상품에 붙어있는 바코드를 읽어들이는 바로 그 시점에 재고량이 조정되고, 신용조회 등 판매와 관련되어
필요한 일련의 조치가 한번에 모두 이루어지는 시스템이다. 

이를 위하여 POS 시스템은 바코드리더, 광학스캐너 카드리더 등이 계산대와 결합되어 있는 PC나 또는 특별한 단말기를 사용한다.
POS 시스템은 신용조회나 재고량 조정 등을 위해 중앙컴퓨터와 온라인으로 연결되거나, 또는 일괄처리를 위해 
주전산기에 전송되기 전까지 일일거래를 저장하기 위해 독립된 컴퓨터를 사용할 수도 있다.

▶ 무전표시스템의 더존 연동을 위해 필요한 사항

1. 법인카드 가상계좌 설정방안
   - 법인카드 결재대금을 급여통장으로 입금할 경우 자동이체등에 의한 금액부족으로 카드대금 연체가 발생할 수 있습니다.
   - 따라서, 법인카드 결재대금은 가상계좌를 이용할 것을 권장하며, 카드신청 시 개인별 가상계좌설정 가능여부를 확인하셔야 합니다.

   - 그리고 각 개인별 가상계좌번호가 추가될때 일부 사용자는 여러개의 법인카드 또는 공용카드를 소지할 수 있으므로 
     사원별 계좌에 추가할 경우 어느계좌가 어느법인카드인지 구분할 수 있는 방법이 필요하게 됩니다.

    - 사원별 계좌에 추가할 경우 법인카드관련 각종 통계 시 타 내역과 섞여 원하시는 리포트를 뽑기 어려울 수 있으니 
      법인카드별로 거래처를 별도생성하시는 것을 권장해드립니다.

     예) 사원거래처 조상욱/코드 사번, 법인카드거래처 조상욱_현대1234 (카드 뒤 4자리)/코드 사번+일련번호

2. 예산코드 및 예산등록
     - 예산코드체계를 확정하고 일부예산이라도 등록해야 예산통제관련 작업을 진행할 수 있습니다.

3. 결의코드 작성
     - 각종 전표유형을 도출하여 결의코드를 작성해야 전표입력화면을 개발할 수 있으며 전표 연동테스트가 가능해집니다.
     - 더존에서는 결의코드 작성에만 2주이상 걸릴 수 있다고 하니, 일단 한두개의 유형만이라도 등록하여 연계테스트가 가능해지도록 요청드립니다.

4. 코스트센터 정리
     - 종료된 코스트센터의 종료일을 확정해야 중복된 코스트센터가 나오지 않도록 할 수 있습니다.
